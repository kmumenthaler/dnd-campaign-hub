
# Templates & Migration

Purpose

Templates drive the autogenerated note content (PC/NPC/item templates, stat blocks, etc.). The migration system ensures existing notes are updated safely when templates change.

Where to look

- Templates: [src/templates.ts](src/templates.ts)
- Migration manager: [src/migration/MigrationManager.ts](src/migration/MigrationManager.ts)
- Migration modal/helpers: [src/migration/MigrationModal.ts](src/migration/MigrationModal.ts)

Rules & checklist

- Increment the `template_version` inside the template string whenever you change displayed or inserted content.
- Update `TEMPLATE_VERSIONS` in `MigrationManager.ts` to match the new version.
- Add migration functions for non-trivial changes that add new blocks or UI (never remove user data).
- Make migrations idempotent: check whether the new content already exists before inserting.

Minimal migration pattern

1. Add a migration method in `MigrationManager` (e.g. `migratePCTo1_2_0(file: TFile)`).
2. In `migrateFile()`, call the migration when `compareVersions(current, "1.2.0") < 0`.
3. At the start of the migration, read the file and check for a marker string (so repeated runs are safe).
4. Insert only the additional template content and finally update the `template_version` in the file's frontmatter.

Example (pseudocode)

// inside MigrationManager
async migratePCTo1_2_0(file) {
	const content = await this.app.vault.read(file);
	if (content.includes('<!-- pc-edit-controls -->')) return;
	const newContent = content.replace('\n---\n', '\n---\n' + '...new template block...\n');
	await this.app.vault.modify(file, newContent);
	await this.updateTemplateVersion(file, '1.2.0');
}

Testing & deployment

- Build and run the plugin locally and test migration on a copy of a campaign (migrations write backups to `z_Backups/`).
- If a migration touches many files, consider adding a migration progress modal or breaking it into smaller steps.
